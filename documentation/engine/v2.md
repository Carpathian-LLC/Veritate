# engine v2

`veritate_engine/v2/` is an empty scratchpad reserved for future engine experiments.

## intent

When a future engine change is large enough that it cannot be folded into `v1/` without altering the hot path or breaking format compatibility, the experimental version goes here as a sibling tree. v1 stays pristine; v2 carries the new work until it stabilizes.

The current engine fits entirely in `v1/`. v2 is empty. Adding ternary support (`.bin` v10) did not require a sibling engine: it landed as a load-time format reader (`load_b_ternary` in v1's `model.c`), so the v1 hot path stayed unchanged.

## when to populate v2

- A new quantization scheme that changes the runtime matmul (not just the disk format).
- A different attention or block topology that v1 cannot represent without a hot-path branch.
- An ABI break (changed activation scale convention, residual stream width, etc.) that would otherwise break every existing `.bin`.

A pure load-time format reader, a new `.bin` version, or a per-arch kernel optimization is not a v2 candidate; those go into v1 with the existing format-version dispatch and per-arch kernel pattern.

## layout when populated

```
veritate_engine/v2/
├── src/
├── kernels/{scalar,arm64,x86_64,inline}/
├── build/
└── bin/<os>/<arch>/veritate_v2
```

The build script outputs a binary named `veritate_v2` so it can sit alongside v1 in the same `bin/<os>/<arch>/` tree without collision. `paths.py` gains an `engine_version` argument when v2 is alive.

## status

Empty. Folder exists as a marker; no source, no binary, nothing to build.
