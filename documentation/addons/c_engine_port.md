# C engine addon contract

Inference-time addons for the C engine. Mirrors the python contract in [addons/contract.md](contract.md): three methods (`reset`, `observe`, `bias_logits`), one chain that composes them, no model weights touched. The C side adds a `destroy` method for explicit teardown since C has no garbage collector.

This is a versioned contract. Adding, removing, or changing the signature of any function or field below requires updating this file in the same commit.

# ------------------------------------------------------------------------------------
# Why a separate contract

The python contract lives at the sampler boundary in `Brain.stream`. The C engine has its own sampler in `model.c::sample_token_ext`. Mirroring the python contract in C lets the same addon idea (slot table, future v2 addons) attach to either backend. They are not the same code -- they are the same **contract**, expressed twice. Bit-parity between the two implementations is the test that the contract held.

Lookups, regexes, and string helpers that the python addon takes for granted are hand-rolled in C. Each port is a careful translation. The slot_table port is the canonical example.

# ------------------------------------------------------------------------------------
# vtable

```c
typedef struct addon_t addon_t;

typedef struct {
    void (*reset)      (addon_t* self);
    void (*observe)    (addon_t* self, int byte_int);
    void (*bias_logits)(addon_t* self, float* logits, int32_t vocab);
    void (*destroy)    (addon_t* self);
} addon_vtable_t;

struct addon_t {
    const char*           id;
    const addon_vtable_t* vtable;
    void*                 state;
};
```

| method | when called | what it does |
|---|---|---|
| `reset` | start of each new prompt (engine calls before any `observe`) | clear all internal state. |
| `observe` | once per byte (prompt + sampled) | mutate state from the byte that just landed. |
| `bias_logits` | once per step, before sampling | mutate `logits` (length `vocab`) in place. |
| `destroy` | when the chain is freed | free `state`. The chain frees the `addon_t*` itself. |

`logits` is a `float*` of length `vocab` (always 256 in the byte-level engine). The bias is applied in float space; the engine converts int32 logits to float, runs the chain, and converts back. NEG_INF (`-1.0e30f`) marks blocked bytes; addons should preserve the convention.

`observe` runs after each byte commits. Addons do not see un-sampled candidates.

`destroy` runs once per addon at chain teardown. Implementations free their state; chain code frees the `addon_t` struct itself.

# ------------------------------------------------------------------------------------
# chain

```c
typedef struct {
    addon_t** addons;
    int32_t   count;
    int32_t   cap;
} addon_chain_t;

void addon_chain_init   (addon_chain_t* c);
void addon_chain_free   (addon_chain_t* c);
void addon_chain_add    (addon_chain_t* c, addon_t* a);
void addon_chain_reset  (addon_chain_t* c);
void addon_chain_observe(addon_chain_t* c, int byte_int);
void addon_chain_observe_bytes(addon_chain_t* c, const uint8_t* bs, int32_t n);
void addon_chain_bias_logits(addon_chain_t* c, float* logits, int32_t vocab);
```

The chain pipes `bias_logits` through addons in addition order. `observe` and `reset` are broadcast. NULL chain or empty chain is a no-op (zero overhead, identical to the pre-addon engine).

# ------------------------------------------------------------------------------------
# global registry

```c
extern addon_chain_t* g_addons_chain;
void addons_set_global(addon_chain_t* c);
addon_chain_t* addons_get_global(void);
addon_chain_t* addons_build_chain(const char* csv);
```

`g_addons_chain` is read by `sample_token_ext` once per generated byte. It is set once at process startup from `VERITATE_ADDONS=<csv>`, parsed in `main()` before any subcommand dispatches. The CSV is a comma-separated list of addon ids; unknown ids log a warning and are skipped.

The MRI server populates the env var when spawning the C subprocess: a future commit threads the `?addons=<csv>` query param into the subprocess environment. Until then, set the env var manually for engine-side testing.

# integration point

```c
// model.c::sample_token_ext
addon_chain_t* chain = addons_get_global();
if (chain != NULL && chain->count > 0) {
    for (int32_t v = 0; v < V; v++) fp[v] = (float)logits[v];
    addon_chain_bias_logits(chain, fp, V);
    for (int32_t v = 0; v < V; v++) {
        if (fp[v] <= -1.0e30f) logits[v] = -2147483647 - 1;
        else                   logits[v] = (int32_t)fp[v];
    }
    /* recompute argmax over biased logits */
}
```

```c
// main.c::chat_traced_loop
addon_chain_t* g_chain = addons_get_global();
if (g_chain != NULL && g_chain->count > 0) {
    addon_chain_reset(g_chain);
    for (int32_t i = 0; i < n; i++) addon_chain_observe(g_chain, tokens[i] & 0xFF);
}
/* per-byte loop */
if (g_chain != NULL && g_chain->count > 0) addon_chain_observe(g_chain, (int)b);
```

The trace export (`out_logits` in `sample_token_ext`) captures the **unbiased** int32 logits so the dashboard's trace view never lies about what the model produced. Addons only redirect sampling.

# ------------------------------------------------------------------------------------
# adding a new addon

1. Drop `veritate_engine/src/addons/<id>.c` next to `slot_table.c`.
2. Implement the four vtable functions plus a public factory `addon_<id>_new(void)`.
3. Add a forward decl for the factory in `veritate_engine/src/addons.h`.
4. Add a row `{ "<id>", addon_<id>_new }` to `ADDON_FACTORIES` in `addons.c`.
5. Add the `.c` to `veritate_engine/build/build.bat` and `build.sh`.
6. Test bit-parity against the python addon by running both with the same prompts and a fixed RNG seed; sampled byte sequences must match while both backends are healthy.

The C addon must produce the same biases as its python counterpart (bytewise). Tolerances: float32 logit space, so accept difference <= 1e-5 per byte. The sampler choice is identical when biased argmax is identical.

# ------------------------------------------------------------------------------------
# what the contract does NOT cover

- C-side manifest discovery (the python addons read `manifest.json` at runtime). The C engine compiles addons in via the static factory table and uses CSV ids only. Adding manifest discovery would mean shipping a JSON parser to the engine; not worth it until a use case demands it.
- Per-request param overrides. The python `/generate?addons=<csv>` path defaults to manifest defaults; the C engine inherits the same limitation. Future overrides would live in a separate CSV format like `slot_table:pronoun_penalty=6`.
- Engine-only addons (telemetry, instrumentation). Out of scope; instrumentation already lives in the trace path.
- Multi-threaded chain mutation. Each chain is single-threaded; the engine's sampler runs on one thread per generation.

# ------------------------------------------------------------------------------------
# stability

Method names, signatures, vtable layout, and chain functions are stable across patch releases. Renames are minor-version bumps and require this file's update obligation in the same commit.

# ------------------------------------------------------------------------------------
# update obligation

Adding, removing, or renaming any field above requires:

1. The implementation in `veritate_engine/src/addons.c` is updated.
2. The header in `veritate_engine/src/addons.h` is updated.
3. The integration in `veritate_engine/src/model.c::sample_token_ext` and `main.c` is updated.
4. The build scripts (`veritate_engine/build/build.bat`, `build.sh`) are updated.
5. This file's tables are updated in the same commit.
6. Every shipped addon (`veritate_engine/src/addons/*.c`) is verified to still compile and produce the same biases as its python counterpart.
