# hybrid chat (front-door chat endpoint)

## What it is

The chat endpoint behind the `/` front door. Lives at [veritate_mri/routes/hybrid_routes.py](../../../veritate_mri/routes/hybrid_routes.py). It answers a user message with a model the user selects: a locally trained model on the PyTorch or C engine, the public Carpathian model, a configured teacher provider, or an installed Ollama sub-model. Any of them can carry multi-turn conversation context (with automatic compaction) and be grounded with the local knowledge base. Served to users at `/` and `/chat` (page `veritate_mri/web/hybrid.html`).

## How it works

`POST /hybrid/chat` resolves the request body: `model`, `backend` (`pytorch` | `c`), `use_rag`, and the conversation memory (`history` + `summary`).

1. **Routing.** `_resolve_route(model, backend)` returns a uniform `complete(messages, system) -> text` for the selected model, plus its label, response-backend, and kind (`remote` | `local`). `teacher:<provider>:<model>` routes to that provider/model (the model name may contain colons; the id is split on the first colon after the prefix). The bare `teacher` id routes to the active teacher in Settings (legacy). `is_local_model(name)` ([hybrid_routes.py](../../../veritate_mri/routes/hybrid_routes.py)) runs a locally trained model. Anything else (empty, `cloud`, unknown) is the public Carpathian model. A route raises `ChatUnavailable` for an expected failure (bad key, unreachable host, nothing configured), mapped to 503.
2. **Retrieval (opt-in), every model.** When `use_rag` is set and the knowledge base has files (`has_corpus()`): BM25 over the corpus returns the top-`k` chunks (`TOP_K = 3`), each capped at `KB_CHUNK_PREVIEW` bytes. Pure local lexical search, no embedder, no external service. `kb_scope` selects which corpus: `all` (platform + user, default), `platform` (shipped Veritate docs only), or `user` (uploads only). Facts ground **every** route, not just local models: `_system_text` frames them in the system prompt for remote models and as a `context:` block for local byte models. The chat-page **Veritate (platform docs)** pick is the special model id `veritate_docs`: the route maps it to the public model with `use_rag` forced and `kb_scope="platform"`, labelled `Veritate (platform docs)` (it is not a backend model of its own). Recent Training-tab log lines (`training_log_lines()`: in-memory ring entries from sources `plugin`, `plugin:<id>`, `save`, tailed to `TRAIN_LOG_TAIL`) are folded into the grounding whenever `use_logs` is set or in `veritate_docs` mode, so the assistant can answer questions about a run. The chat page exposes this as a **Read my recent training logs** checkbox.
3. **Conversation memory + compaction.** The client carries the conversation as `{summary, turns}` and sends it each turn. After answering, `_compact` folds older turns into the running summary once the memory's characters exceed the selected model's budget (`context_limit_chars`), keeping the last `CTX_KEEP_TAIL_TURNS` verbatim; the summary is generated by the same selected model (`SUMMARY_SYSTEM`). The updated memory plus a `context` gauge (`_context_meter`: `{turns, chars, char_limit, pct}`, `pct` 0..1) are returned for the client to store and render. Compaction is best-effort: a failed summarize leaves history intact.

   **Model-aware budget.** `context_limit_chars(kind, provider, model)` sizes the memory budget to the model: a local byte model uses its trained `seq` (chars == bytes, e.g. 1024); cloud/teacher use a nominal per-provider window (`CONTEXT_TOKENS_BY_PROVIDER` × `CHARS_PER_TOKEN`), scaled by `CONTEXT_MEMORY_FRACTION` to reserve room for the preamble, facts, question, and reply. The OpenAI-compatible gateway does not report a model's real window, so cloud/teacher values are nominal model-family sizes and a host's actual served limit (e.g. an Ollama `num_ctx`) may be lower.
4. **Generation.** `complete(messages, system)` runs the selected route:
   - Remote (cloud/teacher): a messages list (`history` + the new user turn) plus the system preamble. Cloud goes through `ai_assist.chat`; teachers through `teacher.client.complete` with that provider's key + base url from `teacher_configs[provider]` (falling back to the flat `teacher_*` slots and `VERITATE_TEACHER_API_KEY` for the active provider).
   - Local: `_render_local` builds a multi-turn byte prompt (prior turns in chat markers, the final turn via `build_prompt` / `build_plain_prompt`); `_ensure_pytorch` / `_ensure_c` load the model and `_generate_local` decodes via `brain.stream` / `_c_engine_stream`. `collect()` joins the `token` / `fast_byte` bytes; `_trim()` cuts at the first stop marker (`<|end|>`, `<|user|>`, `\ncontext:`).

The public model key is the `PUBLIC_AI_KEY` constant in [settings.py](../../../veritate_mri/runtime/settings.py), injected live by `settings.get()` (never persisted, so rotating it reaches every install on next load); a user-set `ai_api_key_user` overrides it. Generation reuses platform code; this module owns no decode loop and pins no model name. The corpus chunks + BM25 index are cached in `_STATE` and rebuilt on the next retrieve after an upload (`reset_index_cache`). The loaded model lives in shared app config (`cfg["BRAIN"]` / `cfg["C_SUBPROCESS"]`), the same state the platform backends use.

### Knowledge base

Retrieval spans two folders (`_corpus_files()`): the shipped `PLATFORM_KB_DIR` (`veritate_mri/data/platform_kb`, committed to the repo) and the user's `KB_DIR` (`veritate_mri/data/kb`, gitignored uploads). The platform KB carries the Veritate documentation, so a fresh install can ground answers about the platform with no upload; `has_corpus()` is true whenever either folder has files. `kb_files()` counts user uploads only (the upload UI's file count). `_kb_chunks()` reads every file across both folders and splits it with the platform retriever's whitespace-aware `_split_chunks`; `_kb_index()` builds a `BM25Index` ([inference/agent/tools/retriever.py](../../../veritate_mri/inference/agent/tools/retriever.py)) over those chunks. No Ollama, no embeddings, no `.npz`. To revise the platform answers, edit the markdown in `platform_kb/`.

### Chat page UI

The page ([veritate_mri/web/hybrid.html](../../../veritate_mri/web/hybrid.html)) is a self-contained dark page with the platform `V·E·R·I·T·A·T·E` logo. A `settings` toggle opens an inline panel (with a **Close settings** button along its bottom, `#settingsClose`) holding three sections:

- **Model** select: grouped `optgroup`s. A `Veritate` group with **Veritate (platform docs)** (value `veritate_docs`, the default selection for a fresh visitor) — chat grounded on the shipped platform KB. Then `Local (trained)` from `/pytorch-models`; then one group per remote source from `/hybrid/models` — `Cloud` (the `Carpathian AI (public)` entry, value `cloud`), plus a group per configured teacher provider. API providers contribute their saved model; local providers (Ollama, LM Studio, llama.cpp) are enumerated live so every installed sub-model is listed. Each remote entry's value is its `teacher:<provider>:<model>` id. A `Train your own →` link points to `/app#training`. Selecting a remote model hides the engine section; the `veritate_docs` pick also hides the knowledge section (its grounding is implicit). A spinner (`.spin`) shows in the pending reply bubble until the (non-streamed) response returns.
- **Engine** segment: `PyTorch` / `Veritate (C)`. The C button is disabled for models without a `veritate.bin` (cross-checked against `/c-models`), with an inline hint.
- **Context** section (shown for every model except the `veritate_docs` pick, which grounds implicitly): **Use my uploaded files when answering** (`#ragChk`, disabled until the corpus has files), **Read my recent training logs** (`#logChk` → `use_logs`), and a file uploader that posts to `/hybrid/kb/upload`. Upload is instant (no embedding); the page then refreshes `/hybrid/health` and enables grounding.

A header **new chat** button clears the conversation memory and the transcript. A small SVG **context ring** (`#ctxArc`, with the percentage in its center) sits in the chat input bar and fills toward the model's context budget from the response's `context.pct`; clicking it opens a popover (`#ctxPop`) explaining the gauge and showing the live `chars / char_limit`. It is driven entirely by the server meter (no client-side estimate), so it reads 0% until the first reply of a session. Opening settings blurs the chat behind it (`#settings.open ~ .wrap`). A **spinner** shows in the pending bubble while the (non-streamed) reply is in flight. **Slash commands** (`handleSlash`): `/new`, `/model <name>`, `/rag on|off`, `/help` — parsed in-page, no backend call. Settings persist in `localStorage` under `veritate_chat_settings_v1`; the conversation memory `{summary, turns}` persists under `veritate_chat_convo_v1`. Each send posts `{message, model, backend, use_rag, kb_scope, history, summary}` to `/hybrid/chat`, replaces the stored memory with the response `memory`, updates the ring from `context`, and renders `sources` only when retrieval returned any.

## Endpoints

- `GET /hybrid/health`: returns `{ok, has_corpus, n_files}`. The chat page uses `has_corpus` to gate the retrieval checkbox.
- `GET /hybrid/models`: returns `{models: [{id, label, group, provider?, model?}]}` — the public model plus every configured teacher's selectable models (local providers enumerated live with a short `ENUM_TIMEOUT_S` so a down host can't stall the picker). Fetched in parallel with the model lists on page load.
- `POST /hybrid/kb/upload`: multipart `file`. Saves it into `KB_DIR` (sanitized name, `.txt` default), clears the index cache, and returns `{ok, filename, n_files, n_chunks}`. Rejects no-file with 400 and oversize (> `UPLOAD_MAX_BYTES`) with 413.
- `POST /hybrid/chat`: body `{message, model, backend?, use_rag?, use_logs?, kb_scope?, k?, history?, summary?}`. `model` may be a local name, `cloud`, `teacher:<provider>:<model>`, or `veritate_docs`. `use_logs` folds recent training logs into the grounding. `kb_scope` is `all` | `platform` | `user`. `history` is the prior `[{role, content}]` turns (capped at `HISTORY_MAX_TURNS`); `summary` is the running compaction summary. Returns `{ok, answer, model, backend, confident, sources, memory, context}`. `memory` is the post-turn `{summary, turns}` the client stores for the next request; `context` is the gauge `{turns, chars, char_limit, pct}` (`char_limit` is model-aware); `sources` is `[{text, score}]` (RAG hits carry a score, folded-in training-log lines carry `score: null`). Empty message → 400; an unavailable model (bad key, unreachable host, nothing configured) → 503; unexpected generation failure → 500.

Both are public (auth gate exempts the `/hybrid` prefix and `/`). See [auth.md](auth.md).

## Dependencies

- For local generation: a trained model with a checkpoint (PyTorch) or a `veritate.bin` export plus a built C engine (C). Model lists come from `/pytorch-models` and `/c-models`.
- For retrieval: nothing external. Pure-Python BM25 over the uploaded `KB_DIR` files.
- For the public model: the public Carpathian key constant `PUBLIC_AI_KEY` in [settings.py](../../../veritate_mri/runtime/settings.py) (base `https://api.carpathian.ai/ai/v1`), injected by `settings.get()` and called via [ai_assist.py](../../../veritate_mri/runtime/ai_assist.py). A user-set `ai_api_key_user` overrides it.
- torch and the c engine subprocess (lazy-imported, only for local generation).

## Pitfalls

- BM25 is lexical: it matches on shared keywords, not paraphrase. Retrieval quality comes from the uploaded corpus, not a fancy ranker.
- The C engine option requires a `veritate.bin` export and a built engine binary. The chat page disables C for models without a `.bin` (via `/c-models`); a missing engine binary surfaces as a 503 from `_ensure_c`.
- Switching the selected model swaps the shared `cfg["BRAIN"]` / `cfg["C_SUBPROCESS"]`, the same slots the platform tab uses; the last-loaded model is the current one for both surfaces.
- The public model key is the `PUBLIC_AI_KEY` constant, injected by `settings.get()` and not persisted. Rotating it in source reaches every install on next load; a stale copy can no longer be stranded in `mri_settings.json`. A user-set `ai_api_key_user` still overrides it.
- The corpus + BM25 index are cached in `_STATE`; uploads clear the cache, but editing `KB_DIR` files on disk out-of-band needs a server restart (or another upload) to take effect. To make the chat answer questions about Veritate itself, drop platform docs into `KB_DIR` (`veritate_mri/data/kb`) and turn on grounding.
- Conversation memory is client-held: the chat page sends `history` + `summary` and stores the returned `memory`. The server is stateless across turns. A compaction turn spends one extra model call (on the selected model) to summarize; for tiny local byte models that summary is low quality, so context on local models is best-effort.
