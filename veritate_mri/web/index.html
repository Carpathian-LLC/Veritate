        what these knobs do
        

          
temperature — sharpens (<1) or flattens (>1) the next-byte distribution. 0.6–0.9 for coherent prose; 0.1 nearly deterministic; 1.5+ noisy.

          
top_k — only the K highest-logit bytes are sampleable. k=1 = greedy. k=256 = no filter. Default 40 cuts the long tail without losing variety.

          
max bytes — bytes to generate before stopping. One forward pass per byte.

          
top_k wins ties: top_k=1 is greedy regardless of temperature. Sweep temperature with top_k held to compare variants.

telemetry surprise, uncertainty, latency over time

      

      

        surprise (red) is the byte we sampled.
        uncertainty (blue) is the model's prediction before we sampled.
        latency (gray) is your CPU. Three signals on the same axis. The gap between red and blue is where the sampling logic shows up.
      

      

        read the spikes
        

          
surprise (red)

          
How unexpected was the byte that got sampled? Calculated as -log2(p) where p is the probability the model gave that exact byte. So a byte the model was 100% sure of has surprise 0; a byte the model treated like 1-in-256 has surprise 8. Long stretches of low surprise are auto-pilot, where the model and the sampler both knew what was coming (mid-word letters, common bigrams). Spikes are the moments the output stream zigged when the model's predictions zagged.

          
uncertainty (blue)

          
How spread was the model's prediction across all 256 bytes before we sampled? It's the entropy of the distribution. Zero would be one-hot ("definitely byte X"). Around 8 would be flat ("no idea, could be anything"). Spikes mostly happen at branching points in the text: right after a space (many letters could start a word), after punctuation, at the start of a sentence, or anywhere the model didn't have a clear next move.

          
reading the relationship between red and blue

          
both lowCruising. The model is confident and we picked what it expected. The boring middle of a word.

          
both spikeHonest uncertainty playing out. The model was genuinely torn across several options and we sampled a typical one of them. The model is showing real uncertainty and the sampler is going with the flow.

          
red < blueLucky pick from a wide distribution. The model wasn't sure (distribution was spread), but we happened to pick one of the more probable bytes from inside it. Surprise is about the one byte we drew; uncertainty is averaged over all 256. A specific byte can easily be more probable than the average byte. Common case.

          
red > blueLong tail draw. The byte we sampled was less probable than the typical byte in the model's prediction. Two reasons it happens: (a) temperature flattened the distribution so unlikely bytes got a real chance, or (b) top_k > 1 let us pick a runner-up even when it was unlikely. To make it impossible, set top_k = 1 (greedy decoding). Temperature alone won't lock it out, because near-ties (51/49 between two bytes) can still flip.


          
patterns over time (the shape of a generation)

          
Spikes trend lower as the text gets longer. The dominant pattern across almost every generation. Early on, the model has very little context to work with: just the prompt. Lots of valid continuations exist, so both red and blue spike high. As more bytes accumulate, the context narrows what's plausible. The model commits to a topic, a tense, a character, a sentence rhythm. Each next byte has fewer reasonable options, so the prediction sharpens and the spikes flatten out. By the time you're 100 bytes deep, the model is mostly auto-piloting through clauses it can finish on rails.

          
Periodic bumps that aren't decay. Even mid-generation, you'll see small blue bumps right after spaces, periods, and newlines. These are sentence-level branch points — many words could start the next clause. They don't mean the model is "losing it," they mean the language naturally has more freedom at those positions.

          
The inverse (rising spikes late) is a warning. If uncertainty drifts upward late in generation, the model is starting to lose the thread. Two usual causes: the prompt has scrolled out of the 256-byte context window so it's predicting from generated bytes only, or temperature is high enough that random sampling has steered the story somewhere the model wasn't trained for. If you see this consistently, lower temperature, lower top_k, or shorten max bytes.


          
latency (gray)

          
Wall-clock milliseconds for one forward pass. Not a signal about the model. It's about your machine: background processes, OS scheduling, thermal throttling. Latency also creeps up as the context grows because attention is quadratic in sequence length, so the 100th token takes longer than the 10th. Useful to plot alongside the others so you can rule out "that wasn't the model, that was your machine."

        what's rendered here
        

          
what is a "neuron"

          
Each transformer layer has two halves. The first does attention (mixes information across positions in the sequence). The second is the feed-forward network (FFN), which does per-position computation. The FFN expands the hidden state from 768 dimensions out to 3,072, applies a non-linear function (GELU), and projects back down to 768. Each of those 3,072 dimensions is what we mean by a neuron. There are 12 layers × 3,072 = 36,864 FFN neurons in this model. They're where most of the model's "knowledge" lives.

          
what brightness means

          
For each neuron, we take its post-GELU activation at the current token position and compute the absolute value. Big number = strong firing. We then max-pool groups of 12 adjacent neurons into one bucket, so each row shows 256 buckets instead of 3,072 individual neurons (otherwise the canvas would need to be 3,072 pixels wide). Brightness inside a bucket = the strongest-firing neuron in that group of 12.

          
the brain-region analogy

          
Transformers organize roughly the way mammalian brains do, with rough functional regions across depth. Anthropic's interpretability research has named hundreds of specific circuits in real LLMs. For the 12 layers in this model:

          
L0 - L3Sensory cortex. Bytes, n-grams, basic patterns. These layers fire on raw input features: "is this byte a vowel," "did we just see a space," "are we inside quotes." In the chart above, these rows are tinted cool blue.

          
L4 - L8Association cortex. Combines lower-level features into concepts. Word boundaries, syntax, parts of speech, semantic categories. Most of the "what does this story mean" computation lives here. These rows are tinted warm orange.

          
L9 - L11Prefrontal / output. Decides the next byte. These layers project the accumulated context into a specific prediction. Firing patterns here correlate most closely with what gets sampled. These rows are tinted hot red.

          
patterns worth looking for

          
Sparsity. A trained model fires very few neurons strongly per token. Most of the canvas should be dark. Bright cells are the model's "expert" for the current context. If everything looks uniformly lit, the model is probably untrained or the activation values are saturating.

          
Stable hot zones. Specific buckets that stay bright for many tokens in a row. These are neurons specialized to the current topic, character, or sentence type. If you scrub the timeline (in the Learning tab) you'll see these zones emerge over training.

          
Vertical streaks. A column of brightness across multiple layers. That bucket of neurons is participating in this token's prediction at every level of abstraction. Often happens at high-information byte positions (start of a name, first letter of a key word).

          
Layer-specific churn. Early layers (L0 to L3) tend to change firing patterns rapidly token-by-token because they're tracking surface features. Middle layers (L4 to L8) change less often, holding onto semantic context. Late layers (L9 to L11) flip aggressively at sentence boundaries because that's where the next-byte decision is most up-for-grabs.

      confidence four-component calibrated probability
      
      
      
        
      confidence: —
      
        
        
      
        
      
      
      
      Per-token calibrated probability the sampled byte matches a held-out target. Composite =
         sigmoid(w_M·M + w_E·E + w_L·L + w_S·S + b) where M is logit margin (top minus second over sigma),

         sampled byte, and S is the mean Pearson correlation across consecutive layers of the residual projected
         onto the byte's embedding row. Without a fitted weights file the engine falls back to a placeholder formula;
         see docs/CONFIDENCE_MATH.md and training/calibrate_confidence.py.

        Register-NLL proxy — this measures how well the model has learned the prose statistics of each register, not whether it understands the text. FKGL targets syntactic difficulty (sentence length, syllables), not cognitive load: formulaic academic phrasing often scores lower ppl than short high-entropy Pre-K sentences.

          how this metric is computed
          
      
          where the test corpora come from
          
      
          For each of the 7 bands (Pre-K, K, Elem, Middle, HS, College, PhD) we hold out ~4 KB of grade-labeled byte text in veritate_mri/grade_eval/grade_<level>_eval.bin. Sources are fresh hand-authored passages tuned to each band's prose statistics (sentence length, clause density, register), giving clean bytes from offset 0 with no Gutenberg boilerplate to inflate ppl. Probe text never overlaps with training corpora. Sources live at veritate_mri/grade_eval/sources/; producer: veritate_mri/tools/build_grade_evals.py.
          
      
          what perplexity (ppl) actually measures
          
      
          Perplexity = exp(mean cross-entropy). It is the effective number of bytes the model is choosing between at each step. ppl 256 = uniform random across all bytes. ppl 1.0 = perfect prediction. ppl < 3.0 means the model needs under 1.6 bits to commit to each next byte. Lower is better.
          
      
          why this is a proxy and not a reading test
          
      
          Byte cross-entropy measures how predictable the next byte is, not whether the model has built a representation of meaning. The "PhD" band's prose is full of formulaic genre phrasing ("we present", "p < 0.05") and is often easier to predict than short Pre-K sentences with high local entropy ("Bug. Big bug! Run!"). A model can score well on a band purely by recognizing its register without comprehending the text. Treat this tile as register coverage, not reading ability. The hard "can it read" question is the Deep Eval panel below and the comprehension probe (in progress).
          
      
          how to read the bands
          
      
          Band below the ppl-3.0 reference — the model has learned this register's prose statistics well.
          
      
          Band 1-2 ppl above the reference — close to learned. A few thousand more steps on similar text usually closes it.
          
      
          Band far above the reference — that register is under-represented in the training mix, not necessarily "too hard."

          what this actually measures
          
      
          the task
          
      
          For each grade-band passage, ~30 items each present the model with a passage prefix followed by a slot the model must fill. Four candidates: the correct content word and three distractors drawn from the same passage. The model picks via lowest mean per-byte NLL on each candidate conditioned on the prefix.
          
      
          why this is honest where the fluency tile is not
          
      
          Distractors share the passage's register, genre, vocabulary, and syntax. Length-normalization is matched (distractors are within ±2 bytes of the correct word). The slot sits mid-passage, so sentence-initial bigram cues don't help. The only way to win is to use long-range context to disambiguate meaning. A model that has only learned register statistics cannot rise above chance here.
          
      
          how to read the bands
          
      
          Below 32% (near chance) — the model has no reliable signal on this band's prose. Either the register is rare in training or comprehension hasn't formed yet.
          
      
          32-55% — emerging context use. The model is picking up local semantic cues but not consistent.
          
      
          Above 55% — a real comprehension signal. The model is reliably using passage context to predict content words.
          
      
          limitations
          
      
          This is contextual prediction, not full comprehension — the model is never asked an explicit question about the passage. For true MCQ comprehension on standard benchmarks, run the Deep Eval panel below (MMLU/HellaSwag/IFEval). The two are complementary: this probe runs every checkpoint and gives per-band signal cheaply; Deep Eval is the slow ground-truth check.

      Real benchmark accuracy, not a perplexity proxy. Click run deep eval and the dashboard loads the selected checkpoint through the same PyTorch Brain backend it uses for inference, scores every question/item with byte-level per-byte log-likelihood (the byte-level analog of acc_norm in lm-eval-harness), and writes the JSON report next to your model in models/<run>/eval_deep/. Cached results auto-load when this panel opens.
      Heads up — if a model is currently training on MPS, deep eval will briefly share that device. The button is the only trigger; nothing schedules this.
    
          
    
          how this differs from the register-fluency tile
          
      
          two complementary views of "is the model getting smarter"
          
      
          Register fluency (above) is fast and runs at every checkpoint via the save hook. It scores per-byte cross-entropy on hand-authored grade-labeled prose (Pre-K through PhD). Low ppl = the model's distribution matches that prose's statistics. It is register coverage, not comprehension. A model can score low ppl on "college-level prose" while still being totally wrong on a college-level multiple choice question.
          
      
          Deep eval (this panel) is the comprehension test. The model has to actually rank the right multiple-choice answer above the wrong ones, or follow a verifiable instruction. Slow (5-20 min per suite), runs only when you click the button, returns real numbers everyone reports.
          
      
          random baselines
          
      
          MMLU and HellaSwag are 4-way MCQ → chance = 25%. The panel color-codes accuracy: green > 50% (above chance by a comfortable margin), amber 25-50% (signal exists but weak), red < 25% (below chance — either the harness or the model is broken).
          
      
          what "good" looks like, by scale
          
      
          Rough expected MMLU bands for a well-tuned byte-level model: ~85M reaches 25-28% (barely above chance), ~300M reaches 28-32%, ~800M-1B reaches 30-40%, ~7B reaches 40-50%. GPT-3 175B lands around 43% MMLU / 78% HellaSwag for reference. HellaSwag is harder for byte-level than for token-level models because every byte of the ending has to be predicted verbatim. Don't compare these numbers directly to tokenized leaderboards — cross-reference with MMLU's letter-mode for sanity. The expected band for your loaded checkpoint is shown next to each result row.
          
      
          where the data comes from
          
      
          The repo ships ~20-item sample data for the smoke test. To run on the real ~14k-question MMLU and ~10k-item HellaSwag val sets, download the official releases (instructions in veritate_mri/eval/README.md) and drop the JSONs in veritate_mri/data/eval/samples/. The endpoint accepts data-path overrides in its POST body.

          writing health structural quality of the model's own writing — NOT a measure of meaning
          
    
          
      
          no writing-health data yet
          
    
          
    
          Five mathematical proxies (vocabulary variety, entity tracking, broken pronouns, duplicate words, self-perplexity) computed over generations from three fixed prompts at every checkpoint. These tell you when the model graduates from gibberish to grammar. They cannot tell you whether the story makes sense. For example, "the volcano married a sandwich" would score fine here. "World-knowledge" coherence requires a human reader or LLM to review.
          
    
          how each metric is calculated
          
      
          vocabulary variety (distinct-3)
          
      
          Take the generated text, split into words, slide a 3-word window across it, count unique 3-tuples / total 3-tuples. Fraction near 1.0 = every 3-word window is unique (rich, varied prose). Fraction near 0 = the model is repeating phrases (mode collapse, the most common early-training failure mode).
          
      
          entity tracking (lexical chain density)
          
      
          Of all "content" words (length > 3 characters, filtering out small function words like "the", "and"), what fraction recur somewhere else in the passage? High = the model keeps mentioning the same characters and objects (suggests it's tracking entities). Low = each sentence introduces fresh nouns with no follow-through (a story about disjoint things).
          
      
          broken pronouns (pronoun unbacked rate)
          
      
          Find every pronoun (he/she/it/they/this/that/etc). For each, check if there's a noun-like word (length > 3, not a pronoun) within the previous 50 bytes. If not, the pronoun is "unbacked" — it's referring to something that was never introduced. Counts unbacked pronouns / total pronouns. High = the writer keeps saying "she" without telling you who.
          
      
          duplicate words (repeat rate)
          
      
          Fraction of consecutive identical words. "the the cat" → repeat rate 0.33. Catches the simplest mode-collapse pattern where a model gets stuck on one token.
          
      
          self-perplexity
          
      
          The model's perplexity on its own generation. Low = the model "stands behind" what it just wrote (consistent with what it would predict given the prompt). Rising over training would mean it's drifting away from its own distribution — usually a sign the sampler is grabbing low-probability tokens too often. This number is also gameable by simple repetition (the cat the cat the cat scores low) so read it together with vocabulary variety.
          
      
          what these proxies cannot detect
          
      
          Narrative logic. Causality. Whether a fact is true. Whether characters behave consistently across paragraphs. Whether the world the story describes is internally coherent. There is no math for those without a model that has world knowledge. To measure narrative coherence, the only honest tool is an LLM judge or a human.

          how the concept probes work
          
      
          what a probe is
          
      
          A fixed (preamble, target) pair, e.g. preamble "the small ", target "cat". We run the model on the preamble and read the next-byte log-probability for each of the 3 target bytes. Mean surprise across those bytes (in bits per byte) is the concept score. The 50 canonical pairs live at training/checkpoint_probe.py::CONCEPTS; the schema is locked in docs/GLASS_MODEL_ROE.md.
          
      
          what the bits mean
          
      
          Surprise is −log₂ P(byte). 0 bits = the model was sure. 8 bits = it gave the byte a 1-in-256 chance, basically random.
          
      
          MASTERED< 1 bit. The model has internalised the association: given the preamble, the target token is the obvious next byte. Adding more training data here is mostly redundant.
          
      
          LEARNING1 - 2.5 bits. The target is on the shortlist but not the favourite. Usually resolves with continued exposure to the same domain.
          
      
          STRUGGLING> 2.5 bits. The target is essentially unranked. Two diagnoses: the model truly hasn't seen the association (corpus gap, fix it at the training-data level), or the preamble is ambiguous and any plausible continuation scores low (the probe is borderline).
          
      
          where to look first
          
      
          The biggest-gaps row always shows the 5 concepts the model is most surprised by right now. Cross-reference with the category they cluster in: if they cluster in math, you're waiting on Stage D's Q&A and arithmetic corpus. If they cluster in actions, the verb-rich corpus needs more weight. If they cluster in meta (question / answer / story / end), the Q&A formatting hasn't been seen enough.

          training health plateau detection on val loss
          
    
          
    
          Looks at the last four val measurements, computes per-step deltas, and labels the trend. Six possible states; click below for what each one means and what to do.
          
    
          read the training health states
          
      
          how the labels are computed
          
      
          Take the last four validation measurements. Compute the percent change between each adjacent pair (3 deltas). Compute the mean and the mean-of-absolute-values. Apply rules in this priority order:
          
      
          If mean delta is positive and big enough (above +0.5%) → REGRESSING.
          
      
          Else if all four points are within 1% of each other → PLATEAU.
          
      
          Else if mean delta is sufficiently negative (below -3%) → IMPROVING.
          
      
          Else if every delta is negative but small → SLOWING.
          
      
          Else (mixed signs, no trend) → BOUNCING.
          
      
          Priority order matters: a noisy run that happens to be slowly trending up will get REGRESSING (the more concerning label) before BOUNCING (the more neutral label).
          
      
          what each state means and what to do
          
      
          WARMING UPNot enough data yet. Fewer than 4 val measurements have been logged. The detector waits.
          
      
          IMPROVINGHealthy. Val loss is dropping consistently across the last 4 evals. The model is learning. Keep going.
          
      
          BOUNCINGMixed signal. Val went up and down without a clear direction. Common near a local minimum: the optimiser keeps stepping past the bottom and coming back. Two things to consider: lower the learning rate (smaller steps), or just keep going to see if the average drifts down over more evals.
          
      
          SLOWINGDiminishing returns starting. Each eval is still showing a drop in val loss but the drops are small (under 3%). The model is in late-stage training. PLATEAU is probably next.
          
      
          PLATEAUStop or cool the LR. Val has flatlined across 4 evals (all changes under 1%). The model has converged on this corpus. Continuing wastes compute. Either stop training, or drop the LR sharply (cosine cooldown extended) to extract a final ~1-3% improvement.
          
      
          REGRESSINGConcerning. Val loss is consistently rising. Three usual causes: (1) overfitting (train loss has gone below val and the model started memorising rather than generalising); (2) the learning rate is too high for this stage and the optimiser is bouncing past the optimum; (3) a numerical instability is propagating. Reduce the LR, watch the train/val gap, and if it persists, restart from the last good checkpoint.
          
    
          
  
          

  
          
    
          
      
          loss curve train + val, log-y axis
          
      
          
        
        
          hover the chart to inspect a step
          
      
          
      
          Showing the current training run only. The CSV may have older runs concatenated; those are filtered out so the curve reads as one continuous training. Cyan is train loss, violet dots are val loss. Any split ending in _train (e.g. qat_train, mamba2_train) feeds the cyan series; any _val feeds the violet series, so QAT and base runs share the same two colors.
          
      
          read the loss curve
          
        
          what loss measures
          
        
          Cross-entropy on the next-byte prediction, averaged across the mini-batch, in nats. Lower is better. A random model scores ~5.55 (= ln 256). A well-trained byte-level model on natural language settles somewhere between 1.0 and 0.7 depending on corpus difficulty.
          
        
          train vs val
          
        
          Train (solid cyan line) is computed on the same mini-batch the model just updated on, so it's noisy and slightly optimistic. Val (violet dots) is computed periodically on held-out data the model has not seen. Val is the signal that matters; plateau detection runs on it.
          
        
          Rows from any trainer (base, QAT fine-tune, future modes) share these two colors. The chart matches by split suffix (_train, _val), not by which script wrote the CSV row.
          
        
          healthy patterns
          
        
          Sharp drop in the first thousands of steps: the model is learning byte frequencies and basic bigrams. Expected.
          
        
          Smooth decay through the middle: the model is learning real structure. Expected.
          
        
          Curve flattening late: diminishing returns. The plateau panel flags this on the val series.
          
        
          Train far below val: overfitting. The model is memorizing instead of generalizing. Less common with byte-level training on a big corpus.

          learning rate schedule
          
      
          
        
        
          hover to inspect
          
      
          
      
          The learning rate at each step. The trainer follows a fixed cosine schedule: it ramps up during warmup, hits a peak, and decays smoothly to a minimum by the end of training.
          
      
          read the LR schedule
          
        
          what LR is
          
        
          Learning rate is a multiplier on every gradient update. The trainer computes a gradient (which direction to nudge each weight to make the loss smaller), then does weight_new = weight_old - LR * gradient. Bigger LR = bigger nudges. Smaller LR = smaller nudges.
          
        
          shape of the curve you should see
          
        
          Steep ramp at the start (warmup): the LR climbs from near zero up to the peak over the first few hundred steps. This protects randomly-initialised weights from being smashed by big updates before they have any structure.
          
        
          Long smooth descent (cosine decay): the LR falls along a cosine curve from peak to min_lr by the end of training. The decay is gradual at first, accelerates through the middle, then flattens as it approaches the floor.
          
        
          why this shape
          
        
          Big LR early lets the model find a rough shape fast. Small LR late lets it polish that shape without bouncing past the optimum. The cosine curve is empirically robust; in practice no fancier schedule has beaten it consistently for transformer training.
          
        
          what to watch for
          
        
          If the curve looks discontinuous (sudden jumps) or has multiple bumps, the trainer was restarted with different hyperparams, or the chart is showing concatenated runs. The current-run filter should keep this clean.
          
      
          
    
          
    
          
      
          throughput tokens per second
          
      
          
        
        
          hover to inspect
          
      
          
      
          Tokens the trainer is processing per second of wall-clock time. Higher is faster training. Drops usually mean something else is competing with the GPU.
          
      
          read the throughput curve
          
        
          what tok/s is
          
        
          Computed as batch_size × sequence_length / step_time. The total number of training tokens the trainer ingests per second of wall clock. Includes data loading, forward pass, backward pass, and optimizer step. Higher = faster training.
          
        
          shape of the curve
          
        
          Should be flat: well-tuned training has steady throughput. Each step processes the same amount of data and the GPU runs at consistent utilisation.
          
        
          Slow upward drift early: normal for the first few hundred steps as caches warm and the kernel scheduler stabilises.
          
        
          Sudden drops: something else used the GPU. A game, video encoding, another training job, or a screen recorder. Throughput recovers when that other workload stops.
          
        
          Slow downward drift over hours: usually thermal throttling (the GPU is hot enough that the firmware reduced clock speed) or VRAM fragmentation. Sustained drops below 80% of peak are worth investigating.
          
        
          what affects the absolute number
          
        
          Batch size, sequence length, model size, and quantisation mode all affect tok/s. Bigger batches help (more parallelism per step). Bigger models hurt (more compute per token). The number is most useful as a relative signal — compare to itself over time, not to other runs.
          
      
          
    
          
    
          
      
          gradient norm
          
      
          
        
        
          hover to inspect
          
      
          
      
          How big the gradient update wants to be on each step (before clipping). High early when the model is finding shape, smaller and steadier as training settles. Spikes hint at instability.
          
      
          read the gradient norm
          
        
          what it is
          
        
          The L2 norm of all gradients across the model on a single training step, measured before gradient clipping is applied. One number summarising how aggressively the model wants to update its weights this step.
          
        
          shape of the curve
          
        
          Big early, decaying: the model starts with random weights, the loss is high, and the gradients are large because there is a lot to learn. As the model converges, gradients shrink. Healthy.
          
        
          Settling around 1 to 3 mid-run: typical for well-tuned transformer training. The exact range depends on the model and optimiser; what matters is that it stabilises.
          
        
          Sudden spike: the model hit something pathological in this batch (a hard example, a numerical edge case, or a bug). One spike that gets clipped is fine. Repeated spikes mean reduce learning rate or check the data.
          
        
          Near-zero gradients: the model has stopped learning. Either it converged (good) or it got stuck on a flat region (bad).
          
        
          connection to clipping
          
        
          The trainer clips the gradient norm at a configured ceiling (default 1.0). Anything above that value gets rescaled down before the optimiser step. Tracking the unclipped norm lets you see when clipping is actually doing work versus sitting idle.

          What do you want to do with a model?
          
      
    
          
    
          
      
      
      
      
      
      
    
          
    
          
      
          
        
          Models train in stages. Each stage is its own run.
          
        
          The trainer you pick declares which stage it teaches. Once a stage is trained, it stays trained when you add the next stage on top.
          
        
            
          
          1. autocomplete — byte pretraining. The foundation; required for everything above.
            2. 
          
          2. chat — chat-format SFT on an autocomplete model.
            3. 
          
          3. agent — tool-use SFT on a chat model.
            4. 
        
          
        
          Read the other tabs to pick the action that matches what you want to do next.
          
      
          
      
          
        
          Train a new model from random weights.
          
        
          The trainer you pick determines architecture and the capability tier it teaches (usually autocomplete).
          
        
          Effect. Creates models/<name>/. Loss begins near 5.55 nats. The trainer's tier moves to in_progress, then trained at the final step.
          
      
          
      
          
        
          Resume from a saved model's latest checkpoint.
          
        
          Same trainer keeps training. A different trainer adds a new tier on top: autocomplete to chat to agent.
          
        
          Effect. Appends to the model's history from the latest step. Tiers already trained stay trained; the new trainer's tier moves to in_progress.
          
      
          
      
          
        
          Train a small student model against a larger teacher's outputs.
          
        
          Compresses a capable model to fit the engine's hardware budget. Teacher endpoint is set in Settings.
          
        
          Effect. Same model output as from scratch, but the loss signal comes from the teacher's logits, not raw corpus bytes. The run cannot start without a teacher provider.
          
      
          
      
          
        
          Generate training data with the teacher. No model is trained.

          Why is your model showing a warning?
          
      
    
          
    
          
      
          The C engine runs INT8. Without QAT (quantization-aware training), weights are usually too small to survive INT8 rounding, so the C engine outputs gibberish even when PyTorch is fine.
          
      
          Fix: continue training your model with QAT on, then re-export.
          
      
            
        
          1. Training tab → continue from saved → pick your model.
            2. 
        
          2. Tick INT8 QAT, drop lr to ~1e-5, start. Stop when loss replateaus.
            3. 
        
          3. Training tab → export to .bin → pick the new _qat checkpoint → export.
            4. 
        
          4. Generation tab → pick the _qat model. Warning clears.

          Corpus library 
          
      
    
          
    
          
      
          A corpus is the training data a model reads. Files land at trainers/corpus/<name>_train.bin and stay there until you remove them. Click install to download one; the Training tab's data picker then lists it. chat entries teach a model to chat. agent entries teach it to use tools (train these after a chat corpus). The rest are raw text for base autocomplete training. Pick a size that fits your model: a 10M model overfits a 5 GB corpus; a 1B model wastes time on 1 MB.

SHA-256 fields are optional; if provided, the install will verify them and abort on mismatch.
          
      
      
      
      
      
      
          
        
        
      
          
      
          
        
        
      
          
    
          
  
          

          


          
  
          
    
          
      
          Fork model
